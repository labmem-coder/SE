# 智能充电桩调度计费系统验收演示方案

## 1. 演示目标

验收时重点展示系统不是只支持单个用户的简单流程，而是能够正确处理多用户同时充电请求、快慢充分队列、调度分桩、桩内排队、重复请求拦截、用户上报异常、管理员确认故障、故障重调度、故障恢复和运营报表。

建议演示顺序：

1. 系统启动与管理员/用户登录。
2. 多用户同时提交充电请求。
3. 查看系统如何按快/慢充模式分队列并分配充电桩。
4. 多用户响应叫号，展示有人正在充电、有人在同一桩后排队。
5. 展示非法或异常情况的处理，例如重复提交请求被拒绝。
6. 用户上报充电桩异常，管理员确认故障。
7. 展示故障中断、剩余电量重调度、故障桩恢复。
8. 查看运营报表，说明系统有统计能力。

## 2. 演示前准备

进入项目目录：

```bash
cd project1/src
```

初始化数据库：

```bash
python server.py seed
```

如果本机没有 `python` 命令，可使用：

```bash
./.venv/bin/python server.py seed
```

启动服务：

```bash
python server.py serve
```

或：

```bash
./.venv/bin/python server.py serve
```

启动后打开：

- 前端页面：`http://127.0.0.1:8000/`
- API 文档：`http://127.0.0.1:8000/docs`

注意：多用户演示脚本假设数据库处于干净的 seed 后状态。如果之前已经用同一数据库跑过演示，可能存在未完成请求，脚本再次提交时会触发系统的 409 重复请求保护。验收前建议使用干净数据库或单独的临时演示数据库。

## 3. 一键命令行演示

服务启动后，另开一个终端运行：

```bash
cd project1/src
python demo_acceptance.py
```

如果服务不是默认 `8000` 端口：

```bash
BASE_URL=http://127.0.0.1:8765 python demo_acceptance.py
```

如果本机没有 `python` 命令：

```bash
./.venv/bin/python demo_acceptance.py
```

如果需要重复演示，使用：

```bash
./.venv/bin/python demo_acceptance.py --reset
```

`--reset` 会清空本地演示库中的充电请求、会话、账单、异常上报和故障记录，并把充电桩恢复为可用状态，保留测试账号、车辆和充电桩基础数据。

脚本会自动完成以下操作：

| 步骤 | 操作 | 展示点 |
| --- | --- | --- |
| 1 | 管理员和 4 个用户登录 | 鉴权可用，用户身份区分 |
| 2 | `alice`、`bob`、`dave` 提交快充请求，`carol` 提交慢充请求 | 快慢充分队列，多用户同时请求 |
| 3 | 管理员查询桩状态 | 2 快 + 3 慢共 5 个桩，系统自动分配 |
| 4 | 4 个用户响应叫号 | 有用户进入充电，有用户进入桩内排队 |
| 5 | `alice` 重复提交请求 | 系统返回 409，阻止同一车辆重复活跃请求 |
| 6 | `alice` 上报异常，管理员确认故障 | 用户上报不直接改桩状态，管理员确认后才故障 |
| 7 | 系统处理中断会话和重调度 | 生成中断会话 ID 和重调度请求数量 |
| 8 | 管理员恢复故障桩并查报表 | 故障恢复、运营统计可用 |

## 4. 多用户请求场景说明

建议现场讲解如下：

> 当前系统有 2 个快充桩和 3 个慢充桩，每个桩后有 4 个车位。我们让 4 个不同用户同时提交请求，其中 alice、bob、dave 选择快充，carol 选择慢充。系统会先按充电模式进入不同等待队列，再按“完成时间最短”的策略给每辆车分配合适的桩。

预期现象：

| 用户 | 模式 | 预期处理 |
| --- | --- | --- |
| alice | 快充 | 分配到某个快充桩，确认后开始充电或进入桩队列 |
| bob | 快充 | 分配到另一个快充桩，确认后开始充电或进入桩队列 |
| dave | 快充 | 因为快充用户数多于快充桩数，可能进入已有快充桩后的排队位置 |
| carol | 慢充 | 分配到慢充桩，不受快充队列影响 |

可以强调：

- 快充请求和慢充请求互不插队。
- 同一模式内按排队优先级调度。
- 分配桩时不是随便分，而是估算“等待时间 + 自己充电时间”后选择完成时间最短的桩。
- 桩内 `pileQueuePosition=1` 表示正在充电，`pileQueuePosition=2` 表示排在同一桩后等待。

## 5. 异常情况展示

### 5.1 重复提交请求

操作：

`alice` 已经有进行中的请求时，再次提交新的充电请求。

预期：

系统返回 `409 Conflict`。

讲解：

> 系统限制同一车辆同一时间只能存在一个进行中的充电请求，避免同一辆车占多个队列位置。

### 5.2 用户上报异常

操作：

`alice` 对正在使用或被分配的充电桩提交异常上报。

预期：

系统生成异常上报记录，但充电桩状态不会立刻变成故障。

讲解：

> 用户只是报告异常，不能直接让设备下线。管理员确认后，系统才会把桩标记为故障，这样可以避免误报直接影响调度。

### 5.3 管理员确认故障

操作：

管理员对异常桩执行故障确认。

预期：

- 故障桩状态变为 `fault`。
- 如果该桩正在充电，当前会话被中断。
- 已充电部分生成阶段账单。
- 未完成的剩余电量重新生成请求并参与调度。
- 原先在该桩排队但未开始充电的车辆回到等待队列，并保留原始优先级。

讲解：

> 故障处理兼顾公平和效率。已经充的部分要结算，没充完的部分不能丢失；排队车辆不能因为设备故障被无故插到队尾，所以系统保留原始排队优先级。

### 5.4 故障恢复

操作：

管理员恢复故障桩。

预期：

故障桩重新变为可用状态，并重新参与调度。

讲解：

> 设备恢复后，系统会重新触发调度，让空闲桩继续服务等待车辆。

## 6. 验收时可展示的关键输出字段

| 输出字段 | 含义 | 展示价值 |
| --- | --- | --- |
| `status` | 请求状态，如 `dispatched`、`charging`、`completed`、`cancelled` | 说明请求状态机正确 |
| `queueNumber` | 用户可见排队号，如 `F001`、`S001` | 说明系统按模式生成排队号 |
| `assignedPileCode` | 分配到的充电桩，如 `F1`、`S1` | 说明调度结果 |
| `pileQueuePosition` | 桩内队列位置，1 表示正在充电 | 说明同一桩后的排队情况 |
| `queueLength` | 某桩当前占用车位数 | 说明充电区容量管理 |
| `waitingQueueFast` | 快充等待区人数 | 说明快充等待队列 |
| `waitingQueueSlow` | 慢充等待区人数 | 说明慢充等待队列 |
| `pendingAbnormalReports` | 未处理异常上报数 | 说明用户上报和管理员处理链路 |
| `interruptedSessionId` | 故障中断的会话 ID | 说明故障确实影响了当前会话 |
| `rescheduledRequests` | 故障后重调度请求数量 | 说明系统做了故障重调度 |
| `faultCount` | 报表中的故障次数 | 说明运营统计覆盖故障 |
| `pendingBills` | 未支付账单数量 | 说明账单管理 |

## 7. 现场讲解提纲

可以按下面顺序介绍，每一步都对应系统真实输出：

1. 这是一个前后端分离的充电桩调度计费系统，后端使用 FastAPI，数据库使用 SQLite。
2. 系统初始化后有 2 个快充桩、3 个慢充桩，每个桩后有 4 个车位。
3. 我现在让 4 个不同用户同时提交请求，其中 3 个快充、1 个慢充。
4. 系统按快慢充模式分队列，快充不会和慢充互相影响。
5. 系统按“完成时间最短”的策略给请求分配充电桩。
6. 用户确认叫号后，车辆进入桩内队列；如果桩空闲则立刻开始充电。
7. 当同一车辆重复提交请求时，系统返回 409，防止重复占位。
8. 用户发现异常后只能上报，不能直接改变设备状态。
9. 管理员确认故障后，系统中断当前会话，生成阶段账单，并把剩余电量重调度。
10. 管理员恢复设备后，故障桩重新加入调度。
11. 最后通过运营报表展示总会话数、充电量、收入、故障次数和账单状态。

## 8. 与测试报告的关系

- `docs/test_report.md`：用于证明系统经过测试，包含测试步骤、期望结果和实际结果。
- `docs/acceptance_demo.md`：用于验收现场演示，强调多用户场景和系统对各种情况的正确处理。
- `src/demo_acceptance.py`：用于快速生成多用户演示输出，减少现场手动操作失误。
